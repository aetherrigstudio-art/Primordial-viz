# Deploy rules — Namecheap Stellar Plus

Facts about the host. The deploy walkthrough is in `deploy/DEPLOY.md`; the
manual checklist is the `deploy-cpanel` skill.

## Client-side privacy — keep the *deployed* site free of AI/tooling fingerprints

**Scope (important):** this is about the **deployed website only** — what a
visitor can fetch from `primordial.video` (`index.html` + `src/` + `.htaccess`).
It does **NOT** apply to local dev tooling: `.claude/`, `tools/`, `test/`, skill
scripts, the MCP server, or the **private** git repo are never served, so they're
out of scope — adopt/run whatever helps there.

Within that scope, prefer (don't agonize over) keeping the shipped files clean:

- **Tool-agnostic comments in `index.html` / `src/`** — avoid gratuitous `claude` /
  `anthropic` / `mcp` / `tools/…` / "generated by" references or commented
  generator commands in code a visitor can View-Source. Minor, but easy to keep tidy.
- **No secrets/keys in client code, and no client call to an AI/LLM/MCP endpoint.**
  The MCP server (`tools/mcp/`) is dev-only — fine locally, just never deployed or
  reachable from the site. (The only `fetch()` is same-origin look JSON.)
- **`.htaccess`** disables directory listing (`Options -Indexes`) and denies
  dotfiles (`<FilesMatch "^\.">`) — sensible hardening regardless of AI.
- **Debug globals** (`window.__*`) are harmless data but avoidable noise; gate or
  drop them if convenient. Not a blocker.
- The repo is **private**; commits carry `Co-Authored-By: Claude` trailers, so
  history would reveal AI authorship — just don't make the repo public without a
  scrub. (No action needed while it stays private.)

Rule of thumb: **lock down what's served; relax for everything local.**

## Host profile

- **cPanel / LiteSpeed / CloudLinux LVE.** Static delivery is the ideal fit —
  HTTP/2, `.htaccess` caching/MIME. The app is 100% client-side, so LVE limits
  barely apply.
- **SSH (jailed), cPanel Git Version Control, and cron** are included → a
  git-based deploy is viable alongside File-Manager upload.

## Static upload

- Upload `index.html`, `src/`, and `assets/` into **`public_html`** (File
  Manager, or cPanel **Git Version Control**).
- Place `deploy/.htaccess` at the `public_html` root.

## HTTPS / SSL — annual re-issue

- **Free 1-yr Sectigo PositiveSSL** is auto-installed day one and satisfies
  `getUserMedia` (mic).
- It is **NOT** auto-renewing Let's Encrypt / cPanel AutoSSL. **Re-issue
  annually**, or set up **`acme.sh` over SSH** for hands-off renewal. Put a
  calendar reminder ~11 months out.

## `.htaccess` responsibilities

- Force-HTTPS redirect (mic depends on it).
- `Cache-Control` / `ExpiresByType` for `js` / `css` / `glsl` / `png`.
- Correct MIME: `AddType application/wasm .wasm` if WASM is ever used. (The
  `.htaccess` also carries an `AddType text/plain .glsl` line, but it is
  **vestigial** — shaders ship as `.js` ES modules now, so no `.glsl` files are
  served.)

## Watch the inode cap

- The plan caps at **~300k inodes** (files + folders), not the "unmetered" disk
  label. Don't commit/upload `node_modules` or the full research corpus; keep
  the deployed tree lean.

## Backend — PHP 8 only (scope: the Namecheap host + web path)

- If a contact/upload endpoint is ever needed, write a single **PHP 8.x** file
  (native LSAPI). **Never** Node/Python — they run only via Phusion Passenger
  and fight the EP=30 / 2 GB RAM cap.
- **Scope (ADR-001):** this rule governs the **Namecheap-hosted web path + static
  deploy** — what runs on Stellar Plus and what ships to `primordial.video`. A
  **separate service** (e.g. an AI/RAG backend the Android client calls) may use
  its own stack on its **own infra** (Render/Fly/Cloudflare/VPS — *not* Namecheap,
  which would hit exactly the Passenger limits above), but it must **never** be
  coupled into the gig web path or the static deploy, and must not add runtime
  deps to `index.html`/`src/`. See `docs/decisions/001-backend-rule-scope.md`.
